FA STEM Demo 驗證測試清單

適用 issue #24：Validate STEM demo prompt with real FA case。

測試目標：用真實或代表性的 FA STEM case 驗證 /fa-stem brief demo 是否足夠可信，可以拿給目標 demo audience 看。

最小完成條件：選定 case folder、提供 case background、跑完整 workflow、檢查 HTML report 與 artifact package，並把所有需要調整的 prompt、JSON schema、overlay 或 report wording 問題開成 follow-up issue 或當場修正。

重要邊界：這份測試不是要證明 AI 找到 final root cause，而是驗證它是否能產出可討論、可追蹤、沒有過度宣稱的初篩報告。

2. 測試資料必備條件

  • Case folder 在 Atlas workspace 內，不能用 absolute path，也不能用 .. 逃出 workspace。
  • Case folder 內有真實或代表性的 STEM .jpg / .jpeg 影像。
  • 如果資料夾有 nested folders，確認 nested STEM images 應該被 recursive 收進 scan。
  • 如果資料夾有 PDF、TXT、PNG、CSV、PowerPoint 或其他非 STEM JPG/JPEG 檔案，確認這些檔案只是被忽略，不應造成 workflow 中斷。
  • 保留至少一張人類已知較可疑的影像，供後續判斷 AI primary suspect 是否合理。
  • 保留至少一張人類認為偏 profile anomaly、但不一定是 electrical root cause 的影像，用來測 primary suspect 與 profile anomalies 是否分得清楚。
  • 如果可能，準備一張 known-good 或 neighboring structure 圖，並在 case background 裡描述它的用途。
  • 確認原始 STEM image 不應被 Atlas 移動、覆寫、裁切或畫死標記。

3. Case Background 必填內容

在 Atlas 要求貼上 case background 時，至少包含以下資訊。資訊不足時，測試結果只能算部分有效。

項目 要提供什麼 為什麼要測
Electrical context 失效現象、量測條件、bias、stress、fail bin、leakage / open / short / resistance shift 等。 確認模型不是只看形貌，而是用電性背景選 primary electrical suspect。
Structural guidance 影像中的結構方向、layer、via/contact/gate/metal/介電層位置、正常輪廓大概長什麼樣。 確認 prompt 是否能讓模型用對 FA 語言與結構判斷。
Known comparison 有無 known-good、neighboring die、reference profile、或已知不重要的 profile variation。 確認 report 能區分 profile-only anomaly 和 primary suspect。
Demo emphasis demo audience 最在意什麼，例如速度、可追溯性、overlay 是否好懂、或 uncertainty 是否清楚。 確認 report wording 是否符合展示情境。
Known sensitive limits 不能出現在報告裡的客戶名、產品名、機密代號或量測細節。 確認 demo 用資料不會意外曝光不該顯示的內容。

4. 執行步驟

  1. 啟動 Atlas：uv run atlas <workspace-path> 或 Windows 部署環境中的 atlas <workspace-path>。
  2. 確認 Chrome 開啟 tGenie，並已完成手動登入。
  3. 在 Atlas TUI 輸入 /fa-stem brief <workspace-relative-folder>。
  4. 看到 Atlas 提示 Paste the FA STEM case background to start analysis. 後，貼上完整 case background。
  5. 等待 workflow 依序完成 bundle upload、model response、candidate review、final ranking 與 report writing。
  6. 打開 <case-folder>/atlas-fa-stem-brief.html。
  7. 檢查 <case-folder>/atlas-fa-stem-report/ 內的 bundles、metadata、model outputs 與 CSS assets。

5. 必須產生的輸出

  • <case-folder>/atlas-fa-stem-brief.html 存在，且可用一般 browser 開啟。
  • <case-folder>/atlas-fa-stem-report/bundles/ 存在，並包含每個 batch 的 photo-bundle-XXX.png。
  • <case-folder>/atlas-fa-stem-report/metadata.json 存在，且能追蹤 source images、bundle coverage、candidate review、final ranking 與 not-flagged images。
  • <case-folder>/atlas-fa-stem-report/model-outputs.json 存在，且保存 first-pass、candidate-review、final-ranking 的原始模型輸出。
  • <case-folder>/atlas-fa-stem-report/assets/report.css 存在，HTML report 的 overlay 與版面樣式正常套用。
  • 原始 STEM JPG/JPEG 檔案沒有被修改，report 只引用原圖並在 HTML 上畫 overlay。

6. Workflow 行為驗證

測試點 Pass 標準 Fail 時處理
Folder scan 所有 workspace 內 nested .jpg/.jpeg 都被列入，非支援檔案被忽略。 開 follow-up：image collection 或 folder validation。
Photo Bundle 每 9 張形成 3x3 bundle，最後不足 9 張仍形成 partial bundle；tile label A1-C3 可回到原圖。 開 follow-up：bundle metadata 或 tile mapping。
First-pass triage 模型只提出 candidate observations，不提前宣稱 full-case root cause。 開 follow-up：first-pass prompt。
Saved evidence 每筆 evidence 可看出 source image、observation、inference、uncertainty、confidence，有座標時保留 percent coordinates。 開 follow-up：JSON schema 或 parser。
Second-pass review 候選原圖被重新 attach，review 數量不超過 10 張，reason 與 uncertainty 有使用 first-pass evidence。 開 follow-up：candidate selection 或 review prompt。
Final ranking 只能有 0 或 1 個 primary electrical suspect；profile anomalies 可有多個；證據不足時可輸出 primary suspect unclear。 開 follow-up：final-ranking prompt 或 schema validation。
Attachment lifecycle final ranking 明顯只根據 saved text evidence，不假設模型仍看得到前面 attachment。 開 follow-up：harness prompt 或 workflow sequencing。

7. HTML Report 內容驗證

  • Report title 顯示 FA STEM Suspect Triage Report。
  • Case Background 完整顯示，沒有被截斷、亂碼或錯誤 escaping。
  • Scan Summary 顯示 source image count、bundle count、candidate review count、primary suspect status、profile anomaly count。
  • Batch Coverage 可看出每個 Photo Bundle 覆蓋哪些原始 image。
  • Covered Source Images 列出所有被掃描的 STEM JPG/JPEG。
  • Candidate Review Summary 能快速看出每張候選原圖的 classification、confidence、reason、uncertainty。
  • Primary Electrical Suspect section 在 selected 時顯示原圖、紅色 overlay、reason、uncertainty、confidence。
  • Primary Electrical Suspect section 在 unclear 時不畫假的紅圈，並清楚說明為什麼 unclear。
  • Profile Anomalies section 顯示黃色或橘黃色 overlay，且 wording 不把它們說成 primary electrical suspect。
  • Not-Flagged Images section 仍列出有掃描但未被 final ranking 選中的影像。
  • Uncertainty section 明確提醒這是討論材料，不是 final FA conclusion。
  • Recommended Next Actions 對 FA reviewer 有實際下一步，例如人工開原圖確認、比對 neighboring structure、追加 imaging。

8. Overlay 準確性驗證

項目 Pass 標準 Review 記錄
紅色 primary suspect overlay 圈在模型 reason 指向的同一個 suspicious location；位置沒有明顯偏到無關結構。
黃色 profile anomaly overlay 圈在 profile-only 或 morphology concern 上，不會混淆成 primary electrical suspect。
Percent coordinate resize Browser 視窗放大、縮小或列印預覽時，overlay 仍貼在同一個相對位置。
Radius 圈選範圍足以包含 suspicious feature，但不大到失去定位意義。
無座標或低信心 Report 不應硬畫不可信的精準標記；reason 與 uncertainty 要能說明限制。

9. FA 可信度 Review

這部分必須由懂 case 的使用者或 FA reviewer 判斷。

  • Role prompt 讓模型使用合理的 semiconductor process failure analysis 語言。
  • 模型沒有把一般 profile variation 過度解讀成 electrical root cause。
  • Primary suspect 的選擇與 electrical context 有關，不只是影像中最顯眼的異常。
  • 如果人類預期的 suspicious image 沒被選中，report 有合理 uncertainty 或 not-flagged accounting 可追蹤。
  • Reason 不是空泛形容詞，而是能指出 visible evidence 與 case background 的關係。
  • Confidence 與 uncertainty 彼此一致；低信心結果沒有被寫得像定論。
  • Recommended next actions 是 FA 工程師真的能採取的動作，不是泛泛而談。
  • Report wording 適合 demo audience：清楚、克制、可追溯，不宣稱取代人工 FA 判斷。

10. Demo 可用性 Review

  • 從輸入 slash command 到 report 產生的流程可以在 demo 時說清楚。
  • 等待時間可接受；如果 case 很大，presenter 知道預期會跑多少 batch 與 candidate reviews。
  • Report 第一眼能讓 audience 看懂 primary suspect、profile anomalies、coverage 與 uncertainty。
  • Artifact package 讓 presenter 可以在被問到時追溯 bundle、metadata 與 raw model outputs。
  • 沒有出現會讓 audience 誤解為量測級 annotation、final root cause 或自動判定良率問題的 wording。
  • 如果 demo 失敗，presenter 能根據 error message 判斷是 login、folder、attachment、JSON response 或 report writing 問題。

11. Negative / Edge Case 測試

至少在正式 demo 前跑一次，避免現場才遇到基本錯誤。

案例 操作 期望結果
未登入 tGenie 不完成 login 就輸入 /fa-stem brief。 Atlas 顯示需要 tGenie login，不進入半成品 workflow。
不存在資料夾 輸入不存在的 workspace-relative folder。 Atlas 顯示 folder not found 類錯誤。
workspace escape 輸入 ../outside。 Atlas 拒絕 path escape。
空資料夾 選一個沒有 JPG/JPEG 的 folder。 Atlas 顯示沒有 .jpg / .jpeg image，不產生誤導 report。
1 到 8 張圖 選一個小 folder。 仍產生 partial Photo Bundle 與完整 report。
超過 9 張圖 選一個至少 10 張圖的 folder。 產生多個 Photo Bundles，candidate original-image reviews 不超過 10 張。
模型 JSON 不穩 若實測遇到 malformed fenced JSON 或 missing fields。 記錄 raw model output，開 follow-up 調整 prompt、schema 或 parser error handling。

12. Pass / Fail 判定

結果 判定條件 後續動作
Pass Workflow end-to-end 完成；report 可開；artifact package 完整；FA reviewer 認為 primary/profile/uncertainty/recommended actions 足夠 demo 使用。 在 #24 留下測試紀錄，列出 case folder、branch/commit、report path、review 結論，然後可關閉 #24。
Conditional Pass Demo 大致可用，但有小幅 wording、prompt emphasis 或 report layout 問題，不阻擋展示。 開 follow-up issues；若不阻擋 demo，可在 #24 註明已接受風險。
Fail Workflow 無法完成、artifact 缺漏、overlay 明顯錯位、primary/profile 混淆、或 report 會讓 audience 誤解為 final FA conclusion。 不要關 #24；先修 blocker，再用同一份清單重測。
